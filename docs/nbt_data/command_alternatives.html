Optimization

Unfortunately, any command directly connected to NBT Data, such as /data, @...[nbt={}], execute if data, etc. Are creating big latency issues when used. It creates lags, freezes, unsyncs, unexpected entity jittering and more.

This is the reason why it is always a better practice to use interfaces - commands, that don't directly change NBT data.

/tp Teleportation

Instead of modifying entity's Pos or Rot, when possible, it is recommended to use /tp command.

Example 1:

  • /data modify entity @s Pos[0] set value 15 Set X coordinate to 15.
  • /execute at @s run tp @s 15 ~ ~ Use "at @s" part to make sure entity is moving relatively to itself. If you are sure command will be executed relatively, then it is better to remove the execute statement.

Example 2:

  • /data modify entity @s Rot set from entity @p Rot Copies rotation from a nearest player
  • /execute at @s rotated as @p run tp @s ~ ~ ~ ~ ~ Use "at @s" part to make sure entity is moving relatively to itself. If you are sure command will be executed relatively, then it is better to remove the execute statement.

/item Modifying inventory slots

Instead of modifying Inventory or Items, it is recommended to use /item command. It can modify player's inventory, it modifies without closing any GUI, so players can see it in real time.

Example 1:

  • /data modify block ~ ~ ~ Items[{Slot:0b}] set value {id:"minecraft:diamond", Count:1b} Changing 0th slot to a diamond
  • /item replace block ~ ~ ~ container.0 with minecraft:diamond 1 More elegant way, that will update contrainer in real time, instead of closing player's GUI.

Example 2:

  • /data modify block ~ ~ ~ Items[{Slot:0b}].Count set value 16b Changing 0th slot item count to 16
  • /item modify block ~ ~ ~ container.0 [...] Replace [...] with your Item Modifier path, where you have set_count operation set to 16 Again, more elegant, though does require an extra file, but gives much more functionality
